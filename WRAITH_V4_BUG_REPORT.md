# Wraith Browser V4 — Comprehensive Bug Report
## Tested: 2026-03-20
## Tester: Matt Gates + Claude
## Sites: Greenhouse (Anthropic), Lever (WeRide), Ashby (Ramp), Indeed, LinkedIn

---

## EXECUTIVE SUMMARY

Wraith has made **significant progress** since V3. Many previously-missing Web APIs now exist
in QuickJS (`fetch`, `XMLHttpRequest`, `localStorage`, `HTMLInputElement`, `DataTransfer`,
`File`, `Blob`, `FormData`, `document.forms`). However, **every single interaction tool is
broken** — `browse_fill`, `browse_click`, `browse_type`, `browse_select`, `browse_upload_file`,
and `browse_submit_form` all fail on real job application forms. The root cause is a
**disconnect between the snapshot @ref system and the tool execution layer**.

### Bottom Line
- **Scraping/navigation**: EXCELLENT (unchanged from V3)
- **Form interaction tools**: ALL BROKEN — zero tools can interact with real form elements
- **eval_js workaround**: WORKS for reading/writing values, but no Event support = React-blind
- **Net assessment**: Cannot apply to any job through Wraith alone

### BUG 1: `browse_fill` — NOT_FOUND on valid @ref IDs
- **Severity**: CRITICAL (blocks all text input)
- **Reproduction**:
  ```
  browse_navigate → https://job-boards.greenhouse.io/anthropic/jobs/5065894008
  # Snapshot shows: @e37 [text] "" with label "First Name *"
  browse_fill ref_id=37 text="Matt"
  # Returns: "@e37: NOT_FOUND"
  ```
- **Same on Lever**:
  ```
  browse_navigate → https://jobs.lever.co/weride/.../apply
  # Snapshot shows: @e10 [text] "" with label "Full name ✱"
  browse_fill ref_id=10 text="Matt Gates"
  ```

  The two systems are disconnected.
- **Evidence**: `document.getElementById('first_name')` finds the element fine in eval_js, and
  `.value = 'Matt'` works and reads back. The element EXISTS — browse_fill just can't find it.

## CRITICAL BUGS (QuickJS Engine)

### BUG 8: `Event` constructor missing — blocks ALL React form interactions
- **Severity**: CRITICAL
- **Status**: UNCHANGED from V3
- **Test**:
  ```javascript
  typeof Event         // "undefined"
  typeof InputEvent    // "undefined"
  typeof KeyboardEvent // "undefined"
  typeof CustomEvent   // "undefined"
  ```
- **Impact**: Cannot dispatch `input`, `change`, `focus`, `blur` events. React-controlled
  forms (Greenhouse, Lever) will never see value changes even when `.value` is set correctly.
- **Workaround test**: Polyfilling Event + calling dispatchEvent works:
  ```javascript
  function Event(type, opts) { this.type = type; this.bubbles = opts && opts.bubbles; }
  window.Event = Event;
  var el = document.getElementById('first_name');
  el.value = 'Matt';
  el.dispatchEvent(new Event('input', {bubbles: true})); // Returns: true
  ```
  **dispatchEvent accepts the polyfill** — so if Event constructors were added natively,
  the event system would work.

### BUG 9: `dispatchEvent` inconsistently available
- **Severity**: HIGH
- **Test**:
  ```javascript
  var el = document.getElementById('first_name');
  typeof el.dispatchEvent    // "function" ← WORKS on elements from getElementById
  typeof document.body.dispatchEvent  // "undefined" ← BROKEN on document.body
  ```
- **Root cause**: `document.body` is a different kind of object than elements returned by
  `getElementById`. Inconsistent prototype chain.

### BUG 10: `querySelectorAll('*')` returns 0 elements
- **Severity**: HIGH
- **Test**:
  ```javascript
  document.querySelectorAll('*').length       // 0 ← BROKEN
  document.querySelectorAll('input').length    // 22 ← WORKS
  document.getElementsByTagName('input').length // 22 ← WORKS
  document.querySelector('#first_name')        // found ← WORKS
  ```
- **Root cause**: The `*` universal selector is broken. Tag-specific and ID selectors work fine.

### BUG 11: `querySelectorAll` attribute selectors broken for `type`
- **Severity**: MEDIUM
- **Test**:
  ```javascript
  document.querySelectorAll('input[type="text"]').length  // 0 ← BROKEN
  document.querySelectorAll('input[name="name"]').length  // 1 ← WORKS (via name)
  // But individual inputs DO have type="text" via getAttribute('type')
  ```
- **Root cause**: The `[type="..."]` attribute selector doesn't match, even though
  `getAttribute('type')` returns the correct value.

### BUG 12: `parentNode`, `children`, `childNodes` missing on elements
- **Severity**: MEDIUM
- **Test**:
  ```javascript
  var el = document.getElementById('first_name');
  typeof el.parentNode  // "undefined"
  typeof el.children    // "undefined"
  el.childNodes.length  // throws or -1
  ```
- **Impact**: Cannot traverse DOM tree. Breaks any code that walks parents/children.

### BUG 13: `browse_fetch_scripts` finds no scripts
- **Severity**: HIGH
- **Reproduction**:
  ```
  browse_navigate → https://job-boards.greenhouse.io/anthropic/jobs/5065894008
  browse_fetch_scripts
  # Returns: "No external scripts found or fetchable."
  ```
- **Reality**: Greenhouse pages have multiple `<script>` tags including React bundles.
  The tool either can't find them or can't process them.
- **Evidence**: `document.querySelectorAll('script').length` — would test but `*` selector
  issues may affect this too.

### BUG 14: Still missing APIs
- **Severity**: MEDIUM
- **Status**: These remain `undefined`:
  ```
  MutationObserver     — "undefined" (needed for watching React re-renders)
  getComputedStyle     — "undefined" (needed for visibility checks)
  requestAnimationFrame — "undefined" (needed for React rendering)
  ```

## ROOT CAUSE ANALYSIS

### The Core Problem: Two Disconnected DOMs

Wraith has **two separate DOM representations** that don't talk to each other:

1. **Rust HTML Parser DOM** → Generates the snapshot with @ref IDs. This is what you see
   when you call `browse_snapshot`. It correctly parses all HTML elements, labels them,
   and assigns @ref numbers.

2. **QuickJS DOM** → The JavaScript execution environment. Has `document.getElementById`,
   `querySelector`, etc. Can find elements and set values. But it has NO connection to
   the @ref ID system.

When you call `browse_fill @e37`, the tool tries to look up element @e37 in... somewhere.
And that "somewhere" is neither the Rust DOM nor the QuickJS DOM — or if it is the QuickJS
DOM, the @ref→element mapping table is empty/broken.

**This is the #1 bug.** If @ref IDs resolved to actual DOM elements, most tools would work.

### The Stub Problem

`browse_type` and `browse_select` return "Action acknowledged (Sevro stub)" — they're
literally placeholder functions. These need actual implementations.

### The Event Problem

Even if @ref mapping is fixed, React forms need:
1. `Event` / `InputEvent` / `KeyboardEvent` constructors (still missing)
2. Consistent `dispatchEvent` on all DOM nodes (works on getElementById elements, not on body)
3. React script execution (browse_fetch_scripts doesn't work)

## EVAL_JS WORKAROUND (Current Best Path)

Until tools are fixed, the ONLY way to interact with forms is via `browse_eval_js`:

```javascript
// 1. Find element by known attribute
var el = document.getElementById('first_name');
// or: document.querySelector('input[name="name"]');

// 2. Set value
el.value = 'Matt';

// 3. Polyfill + dispatch events (for React)
function Event(type, opts) { this.type = type; this.bubbles = opts && opts.bubbles; }
window.Event = Event;
el.dispatchEvent(new Event('input', {bubbles: true}));
el.dispatchEvent(new Event('change', {bubbles: true}));
```

**Limitations of this workaround**:
- Must know element IDs/names (can't use @ref IDs)
- No file upload (no real File API)
- No form submission (no React XHR without React loaded)
- No radio/checkbox toggling (no click events on React components)
- Tedious — each field requires a separate eval_js call
