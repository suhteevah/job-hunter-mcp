## Bug 2: React SPA Rendering Still Broken (Gap 2) — CRITICAL

This is the P0 blocker. Without SPA rendering, gaps 3, 4, 5, 10 are untestable.

### Reproduction — Greenhouse
```
Tool: browse_navigate
URL:  https://boards.greenhouse.io/embed/job_board?for=figma
```

### Output
```
Page: "" (https://boards.greenhouse.io/embed/job_board?for=figma)
@e1    [html]       ""
@e2    [body]       ""
```

Two DOM nodes. Zero content. The React app bundle didn't execute.

### Reproduction — Ashby
```
Tool: browse_navigate
URL:  https://jobs.ashbyhq.com/ramp
```

### Output
```
Page: "Ramp Jobs" (https://jobs.ashbyhq.com/ramp)
@e1    [html]       ""
@e2    [title]      "Ramp Jobs"
@e3    [body]       ""
@e4-7  [div]        "" (empty)
```

Title tag rendered (server-side `<title>`), but React mount point is empty. No job listings, no interactive elements.

### What Chrome shows
Both pages render 50+ job listings with titles, locations, departments, and apply links. The content is entirely generated by React 18 client-side hydration.

### Root cause (likely)
Same as original Gap 2 analysis — QuickJS can't execute:
- External JS bundles (webpack chunks, dynamic imports)
- `requestAnimationFrame` / `requestIdleCallback` (React scheduler)
- ES modules (`<script type="module">`)
- `MutationObserver`, `IntersectionObserver`

### Impact
Greenhouse + Ashby + Lever forms + LinkedIn + Workday + most modern career pages = ~60% of job application targets are unusable.

## Bug 3: Cookie Import Fails When Chrome Is Running (Gap 7)

### Reproduction
```
Tool: cookie_import_chrome
Args: domain="indeed.com"
```

### Output
```
Cannot copy cookie DB (Chrome may be running): The process cannot access the file
because it is being used by another process. (os error 32)
Close Chrome or copy the file manually.
```

### Problem
In real-world usage, Chrome is always running. Users don't close their browser to use Wraith. The cookie DB path may be fixed (Network/Cookies), but the file lock makes it unusable.

### Expected
Either:
- Copy the file using Volume Shadow Copy (VSS) on Windows
- Use SQLite's WAL mode to read without exclusive lock
- Read from Chrome's memory-mapped cookies via Chrome DevTools Protocol
- Fall back to reading Chrome's cookie journal/WAL file

## Bug 4 (NEW): Lever URL Rewrite Appends `/apply`

### Reproduction
```
Tool: browse_navigate
URL:  https://jobs.lever.co/figma
```

### Output
```
Page: "Not found – 404 error" (https://jobs.lever.co/figma/apply)
```

Note the URL changed from `/figma` to `/figma/apply`. Wraith appended `/apply` to the URL before navigating.

### Tested URLs (all rewritten, all 404)
- `https://jobs.lever.co/openai` → `/openai/apply` → 404
- `https://jobs.lever.co/netlify` → `/netlify/apply` → 404
- `https://jobs.lever.co/reddit` → `/reddit/apply` → 404
- `https://jobs.lever.co/figma` → `/figma/apply` → 404

### Expected
Navigate to the exact URL provided. Do not append or modify the path.
